# lazy-project-search.nvim PRD

## 1. Product Positioning

`lazy-project-search.nvim` is a project-level search preset system for LazyVim, powered by Snacks Picker.

The plugin solves a practical problem: every project has repeated search patterns that are meaningful only in that project. Examples:

- React hooks such as `hooks/use-*`
- TanStack Query usage such as `queryKey`, `useQuery`, `useMutation`
- service layer files while excluding tests
- route definitions
- feature module entry files
- Vue `defineProps` / `defineEmits`
- NestJS `@Controller`, `@Injectable`, `@Module`
- TODO/FIXME and environment usage

Instead of repeatedly typing ad-hoc grep commands, users can store those searches as editable JSON presets and execute them from a single picker.

## 2. Current User Experience

Recommended LazyVim entry point:

```text
<C-p>
```

This is configured through lazy.nvim `keys`, not by eagerly loading the plugin:

```lua
keys = {
  {
    "<C-p>",
    "<cmd>ProjectSearch<cr>",
    desc = "Project Search",
  },
}
```

First use in a project:

```text
detect project type
create project rule JSON outside the source tree
open the JSON file for editing
```

Later use:

```text
open Project Search picker
show Search presets first
show Manage actions at the bottom
preview the selected rule on demand
press Enter to run the selected preset
```

## 4. Storage Design

Rules are stored in Neovim's data directory:

```text
~/.local/share/nvim/project-search/rules/<project-hash>.json
```

The hash is based on the project root. This design keeps rule files:

- project-specific
- local to the developer machine
- out of the source repository
- safe from accidental commits

## 5. Rule File Shape

```json
{
  "version": 1,
  "meta": {
    "projectRoot": "/home/lambert/githubRepos/your-project",
    "template": "react",
    "createdAt": "2026-05-20T00:00:00Z",
    "note": "This file is stored outside your project. Edit presets to customize LazyVim project search."
  },
  "presets": [
    {
      "id": "react.query_hooks",
      "name": "TanStack Query: hooks",
      "description": "Search query-related React hooks.",
      "type": "grep",
      "regex": true,
      "search": "useQuery|useMutation|useInfiniteQuery",
      "dirs": ["src"],
      "glob": ["*.ts", "*.tsx"]
    },
    {
      "id": "project.service_files",
      "name": "Service: API layer files",
      "description": "Find TypeScript files under service directories, excluding test files.",
      "type": "files_regex",
      "regex": "service/[^/]+\\.(ts|tsx)$",
      "dirs": ["src"],
      "exclude": ["*.test.ts", "*.test.tsx", "*.spec.ts", "*.spec.tsx"]
    }
  ]
}
```

## 6. Rule Types

### files

Open a file picker rooted at a project directory.

```json
{
  "name": "Files: src",
  "type": "files",
  "cwd": "src"
}
```

Use cases:

- quickly enter `src`
- quickly enter `components`
- quickly enter `routes`
- quickly enter `packages/foo`

### grep

Search inside files.

```json
{
  "name": "React: queryKey",
  "type": "grep",
  "search": "queryKey",
  "dirs": ["src", "app"],
  "glob": ["*.ts", "*.tsx"]
}
```

Use cases:

- `queryKey`
- `className`
- `TODO|FIXME`
- `process.env|import.meta.env`
- decorators such as `@Controller`

### files_regex

Find files by path convention. This is backed by `fd`/`fdfind`, so the regex must be valid Rust regex.

```json
{
  "name": "Hooks: use-* kebab files",
  "type": "files_regex",
  "regex": "(^|/)hooks/use-[a-z0-9-]+\\.(ts|tsx|js|jsx)$",
  "dirs": ["src", "app", "packages"]
}
```

Important constraint:

```text
fd does not support look-around.
Avoid (?=...), (?!...), (?<=...), and (?<!...).
Use exclude for negative filename filters.
```

Example:

```json
{
  "name": "Service: API layer files",
  "type": "files_regex",
  "regex": "service/[^/]+\\.(ts|tsx)$",
  "dirs": ["src"],
  "exclude": ["*.test.ts", "*.test.tsx", "*.spec.ts", "*.spec.tsx"]
}
```

## 7. Picker Design

Current panel layout:

```text
Project Search

Search  Files: src                         files
Search  Search: TODO / FIXME               grep
Search  React: className                   grep
Search  TanStack Query: hooks              grep
Search  Service: API layer files           files_regex

Manage  Edit current project search rules
Manage  Reset current project rules from template
Manage  Copy current rules path
```

Design decisions:

- Search rules appear first because they are the primary workflow.
- Manage actions are kept in the same picker but grouped at the bottom.
- Previews are generated on demand to keep open latency low.
- Empty-query sorting is disabled so the intentional grouping order is preserved.

## 8. Commands

| Command | Purpose |
| --- | --- |
| `:ProjectSearch` | Open project search panel |
| `:ProjectSearchEdit` | Edit current project rules |
| `:ProjectSearchInit` | Initialize rules if missing |
| `:ProjectSearchInit!` | Force regenerate rules |
| `:ProjectSearchReset` | Reset rules from detected templates |
| `:ProjectSearchPath` | Show current project rule path |
| `:ProjectSearchHealth` | Run health checks |

## 9. Installation Model

Recommended LazyVim spec:

```lua
return {
  {
    "CoderLambert/lazy-project-search.nvim",
    main = "project_search",
    dependencies = {
      "folke/snacks.nvim",
    },
    cmd = {
      "ProjectSearch",
      "ProjectSearchEdit",
      "ProjectSearchInit",
      "ProjectSearchReset",
      "ProjectSearchPath",
      "ProjectSearchHealth",
    },
    keys = {
      {
        "<C-p>",
        "<cmd>ProjectSearch<cr>",
        desc = "Project Search",
      },
    },
    opts = {
      keymap = false,
      storage_dir = vim.fn.stdpath("data") .. "/project-search/rules",
      auto_init = true,
    },
  },
}
```

Rationale:

- `cmd` and `keys` keep the plugin lazy-loaded.
- `keymap = false` avoids duplicate key registration.
- `<C-p>` is fast, meaningful, and avoids LazyVim's common `<leader>p` Yank History mapping.
